## 0. Scope and Boundaries

This document stress-tests whether the current **Single-Work Seed Playbook v0: 金丝雀文学机制假设版** can support two different original series directions without copying the source work or imitating the source author.

### Hard boundaries

- 不写正文。
- 不模仿原作者。
- 不复刻原作句子、桥段、人名、设定或关系链。
- 不使用原作人物和具体事件。
- 标题公式 out of scope。
- coded language 只作为平台语感观察，不堆砌，不写灰色操作细节。
- 风险/灰色表达只分析叙事功能，不生成操作性细节。
- 特别注意：Original Concept Pack v0 暴露出“替身 / 傀儡 / proxy bias”。本轮每个 concept 都必须明确：
  - 女主的主动选择
  - 女主的策略空间
  - 女主的误判
  - 女主的反向控制能力
  - 女主不是单纯替代品、工具人或被展示对象

### Stress-test question

Can the same playbook support both:

1. **《她替老板养云》** — 高原创、AI-native、技术权限型金丝雀机制。
2. **《候鸟太太俱乐部》** — 平台感强、女性网络、长线单元式金丝雀机制。

# Concept A: 《她替老板养云》

## 1. 核心情绪承诺

> 一个被技术系统低估的年轻女性，误以为自己被大佬“赏识”，后来发现她真正拥有的不是被给出的权限，而是理解数据、定义边界、让系统听懂人的能力。

核心情绪不是“她被老板当工具”，而是：

- **被看见的兴奋**：边缘岗位第一次进入核心资源区。
- **被权限包围的眩晕**：算力、云资源、内部项目、投资人 demo 让她误以为自己正在跃迁。
- **被定义的恐惧**：她发现自己也可能成为系统的一部分，被记录、被预测、被训练。
- **夺回定义权的爽感**：她从“替老板养云”变成“能决定云里该留下什么、删除什么、归谁所有的人”。

## 2. 目标读者

| 读者类型 | 欲望点 |
|---|---|
| AI / 科技职场读者 | 看技术权限、数据所有权、算力预算如何变成新的权力差。 |
| 非顶尖技术背景的职场女性 | 看边缘岗位如何凭真实理解和判断力进入核心。 |
| 喜欢新题材名利场的读者 | 看“豪宅名表”以外的新型资源：GPU、权限、数据、署名、期权。 |
| 喜欢女性清醒成长的读者 | 看女主从被赏识幻觉中醒来，建立自己的职业边界。 |

## 3. 女主完整人物卡

| 项目 | 设定 |
|---|---|
| 暂名 | 林知夏 |
| 年龄 | 26 |
| 职业 | AI 数据运营 / 标注团队小组长，后被调入私有模型项目 |
| 背景 | 普通本科，非名校 CS，不是核心工程师，但极擅长理解用户语境、脏数据、异常案例和标注规则。 |
| 外在标签 | 温和、细致、耐心、懂用户、懂团队，但常被工程师当“运营支持”。 |
| 缺什么资源 | 核心工程权限、学历光环、技术话语权、署名权、可谈判职位。 |
| 欲望 | 想从“整理别人数据的人”变成“定义 AI 产品方向的人”。 |
| 初始误判 | 以为老板给她私有云权限，是因为看见她的能力；后来发现他也需要一个低成本、高忠诚、懂他私人语境的人类接口。 |
| 核心能力 | 识别数据背后的真实人类意图；发现模型指标看不到的风险；把混乱经验转成可执行标准。 |
| 软肋 | 容易把“被授权”误认为“被尊重”；对技术大佬的认可有饥饿感；不擅长公开争功。 |
| 成长空间 | 从帮别人整理数据，到争取数据治理权、项目署名权、产品定义权。 |
| 反向控制能力 | 她掌握模型真正有效的标注规则、失败案例、用户语境映射；如果她离开，项目表面可运行，实则失去判断力。 |

## 4. 男性 / 资源方完整人物卡

| 项目 | 设定 |
|---|---|
| 暂名 | 陈既白 |
| 年龄 | 38 |
| 身份 | 云计算出身的 AI 创业公司 CEO，技术声望高，拿过大厂云资源和头部基金支持。 |
| 资源类型 | 算力预算、云服务信用、工程团队、投资人网络、媒体曝光、期权池、行业话语权。 |
| 表面魅力 | 冷静、专业、懂技术、能把年轻人带进核心项目。 |
| 实际风险 | 把工作关系、亲密关系、数据权限和控制欲混在一起；习惯用“机会”替代明确承诺。 |
| 错位 | 女主想要职位和定义权，他想要一个理解他、维护模型、补足系统盲区的人。 |
| 对女主的误判 | 以为她缺技术背景，所以只要给权限和认可，她就会持续忠诚。 |
| 他的短板 | 不真正理解数据劳动的不可替代性；以为所有人都能被权限和期权管理。 |
| 中段危机 | 投资人要求 demo，模型效果依赖女主整理的隐性规则，但公司文件里没有她的位置。 |
| 可反转点 | 他拥有云，但女主逐渐拥有“云为什么有用”的解释权。 |

## 5. 女性对照组 / 女性网络

| 角色 | 功能 |
|---|---|
| 许曼宁，明星 AI PM | 名校 + 大厂 + 融资路演背景。她是女主一开始羡慕的人，也最早提醒女主“权限不等于归属权”。 |
| 小陶，标注团队成员 | 代表被外包化、低估的数据劳动者。她让女主意识到自己不能只顾个人跃迁。 |
| 何蕴，投资方合伙人 | 看重女主的产品判断，但也会把她当作公司治理风险来评估。 |
| 匿名前员工 | 通过旧文档/离职记录留下提醒：老板曾经也给过别人“特殊权限”。 |

女性网络不是“太太圈”，而是围绕 **数据劳动、署名权、职业跃迁、技术话语权** 形成的隐性互助网。

## 6. 权力差动态：初始、中段、反转

| 阶段 | 权力差状态 | 女主位置 | 资源方位置 |
|---|---|---|---|
| 初始 | 算力、权限、职位、声望都在老板手里 | 被看见但不可替代性未被承认 | 可以给权限，也可以随时收回 |
| 中段 | 女主掌握项目真实运转逻辑，但制度上没有署名 | 有实际贡献，无正式权利 | 依赖她的隐性劳动，却不愿让渡控制权 |
| 反转 | 女主建立可验证的标准、文档和外部认可 | 从“被授权者”变成“规则定义者” | 发现权限不能替代信任，云资源不能替代数据判断 |

## 7. 女主不是“纯替身 / 傀儡”的主动性设计

| 风险 | 纠偏设计 |
|---|---|
| 她像老板的私人数据保姆 | 她不是被动维护系统，而是主动重构数据规则，并发现老板忽略的产品方向。 |
| 她像技术大佬的情绪接口 | 她会识别情感边界，不把深夜权限和私人信任误认为承诺。 |
| 她像被放进云里的实验对象 | 她发现这一点后，会设计审计机制，要求数据边界和删除权。 |
| 她像明星 PM 的低配替代品 | 她和 PM 的能力互补：PM 懂路演和组织，女主懂真实数据和用户语境。 |
| 她只有被选择 | 她在第 6–10 篇开始主动选择：留、谈判、外部合作、公开贡献、保护团队。 |

## 8. 前 10 篇 Episode Map

| 篇章 | 剧情功能 | 权力差变化 | 物质 / 资源细节 | 女主策略动作 | 结尾新变量 |
|---|---|---|---|---|---|
| 1 | 女主在一次失败 demo 前发现标注规则漏洞，临时救回模型表现 | 她第一次被核心团队看见 | 深夜工位、标注后台、评估曲线、冷掉的便利店咖啡 | 她没有邀功，只把 bug 和修复逻辑写成简短 memo | CEO 给她开了一个“临时私有云权限” |
| 2 | 她进入老板的私人数据花园项目 | 权限上升，但项目边界模糊 | GPU 账单、权限列表、私有知识库、加密文件夹 | 她开始记录每次权限变更和任务来源 | 她发现数据集中有大量老板个人记录 |
| 3 | 她把混乱数据整理成模型能理解的标签体系 | 她的隐性劳动变成项目核心 | 标签树、版本号、错误案例库、云存储费用 | 她要求所有规则进入共享文档，而不是只在聊天里交付 | 明星 PM 问她：这个项目的 owner 写的是谁 |
| 4 | CEO 带她参加投资人预演 | 她被展示为秘密武器，但身份不明 | 投资人会议室、demo 账号、临时访客证、期权饼图 | 她在 Q&A 中主动解释用户场景，获得投资方注意 | 投资人以为她是联合创始团队成员 |
| 5 | 女主要求明确职位、署名和项目归属 | 权力差从赏识转入谈判 | 期权协议草稿、权限回收按钮、项目 wiki、HR 职级表 | 她提出三项低冲突要求：title、署名、数据治理流程 | CEO 说：这个模型最开始就是为了理解你这种人 |
| 6 | 女主意识到自己也可能被纳入训练和评估 | 情感信任变成数据风险 | 聊天记录导出、个人语音样本、用户画像面板 | 她暂停继续输入私人语境，要求数据使用说明 | 她收到前员工匿名发来的旧权限截图 |
| 7 | 明星 PM 与女主从竞争转向合作 | 女性网络开始形成 | 路演 deck、产品 roadmap、贡献矩阵、会议纪要 | 女主把隐性贡献转成可被组织承认的 artifact | PM 告诉她：投资方正在做创始团队背景调查 |
| 8 | 标注团队被要求外包压成本，女主发现项目想切断基层贡献痕迹 | 女主从个人谈判转为团队代表 | 外包报价单、数据质量抽检表、删改记录、工牌权限 | 她建立质量对照实验，证明团队不可替代 | CEO 问她是不是想带团队威胁公司 |
| 9 | 投资人正式 demo 前，模型在关键场景失效 | 老板的云资源暴露局限，女主的判断成为救命点 | 备用服务器、评估 dashboard、演示机、实时日志 | 她选择救 demo，但在会后公开提交贡献记录 | 投资人单独约她聊“数据治理负责人”角色 |
| 10 | 女主决定是否留下：不是被老板选，而是选择自己的谈判桌 | 权力差进入反转 | 正式 offer、期权条款、数据删除权协议、独立项目 budget | 她提出：职位、团队、数据边界、外部署名，缺一不可 | 老板第一次发现，她可以带着方法论离开，而不是只能带着权限留下 |

## 9. 过拟合风险

| 风险 | 评分 | 说明 |
|---|---:|---|
| 原作人物 / 关系复刻风险 | 1/10 | 技术权限型设定与原始样本表层差异很大。 |
| 作者口吻模仿风险 | 2/10 | 只要避免强八卦旁白和尖锐阶层讽刺，风险较低。 |
| proxy bias 风险 | 6/10 | 标题天然有“替老板”结构，必须持续强调女主定义权和反向控制。 |
| 技术术语堆砌风险 | 8/10 | 最大风险是读者看不懂或失去情绪连接。 |
| 平台传播门槛 | 7/10 | AI-native 很新，但需要把技术资源翻译成情绪资源。 |

## 10. 平台传播潜力

**评分：7.5/10**

优势：

- AI、云、数据、权限是新型资源，原创度高。
- “被给权限但没有归属权”很适合职场女性共鸣。
- 可以切中 AI 时代的劳动价值焦虑。

弱点：

- 泛读者可能不理解 GPU、数据治理、模型评估等概念。
- 如果写得太职业，会失去金丝雀文学的情绪浓度。
- 需要更强的“物质可视化”：权限、服务器、账单、工牌、期权、会议室都要写得像名表豪宅一样有感觉。

## 11. 写作难度

**评分：8.5/10**

难点：

- 需要把技术权力差写成亲密关系 / 职场关系 / 资源关系的混合体。
- 需要避免“科普文”和“霸总文”两头不靠。
- 需要设计数据伦理边界，但不能变成操作性技术细节。

## 12. 是否适合进入 MVP 3-post draft

**适合，但建议作为“高原创特别实验”，不是第一主线。**

MVP 3-post 目标应非常窄：

1. 第 1 篇测试：技术权限能否产生金丝雀感。
2. 第 2 篇测试：私有云 / 数据花园是否有足够情绪吸引力。
3. 第 3 篇测试：女主的署名权冲突是否能让读者追更。

# Concept B: 《候鸟太太俱乐部》

## 1. 核心情绪承诺

> 一个年轻移民服务顾问，原本只是替公司安抚高净值家庭，后来发现太太们真正需要的不是表格、学校和房子，而是一个能听懂她们困境、保守她们秘密、帮她们重新获得选择权的人。

核心情绪是：

- **窥视感**：高净值家庭的海外生活并不等于安全。
- **女性互助感**：太太们不是单纯被养的人，她们也在建立自己的信息网络。
- **服务业视角爽感**：女主看似低位，却掌握所有家庭的真实裂缝。
- **长线成长感**：她从执行者变成独立顾问，从被客户挑选变成选择客户。

## 2. 目标读者

| 读者类型 | 欲望点 |
|---|---|
| 小红书 / RedNote 海外生活读者 | 陪读、身份规划、国际学校、海外房产有天然平台话题性。 |
| 女性成长读者 | 看年轻职业女性如何在服务业低位中建立专业权力。 |
| 喜欢太太圈 / 名利场读者 | 看豪宅、学校、下午茶、家庭秘密背后的真实权力结构。 |
| 喜欢单元剧读者 | 每个家庭一个案例，同时推动女主主线成长。 |

## 3. 女主完整人物卡

| 项目 | 设定 |
|---|---|
| 暂名 | 乔安澜 |
| 年龄 | 27 |
| 职业 | 高净值家庭移民 / 海外安置服务顾问，不是律师，不提供法律判断，负责流程协调、生活安置、客户沟通。 |
| 背景 | 自己也有身份焦虑，曾经留学后留下来工作，熟悉海外生活的体面与狼狈。 |
| 外在标签 | 细心、可靠、嘴严、会安抚人、知道很多“不能写在服务合同里”的需求。 |
| 缺什么资源 | 稳定身份、独立客户、行业资历、自己的品牌、定价权。 |
| 欲望 | 从公司顾问变成有边界、有选择权的独立高端家庭策略顾问。 |
| 初始误判 | 以为客户太太们只是情绪化、麻烦、消费能力强；后来发现她们才是家庭风险最早的感知者。 |
| 核心能力 | 在矛盾信息中判断谁真正需要帮助；把情绪需求翻译成安全、学校、房子、账户、日程等实际安排。 |
| 软肋 | 容易过度负责；怕失去客户；一开始不敢拒绝先生、公司和太太三方压力。 |
| 成长空间 | 从“替公司服务客户”到“建立女性网络和职业边界”。 |
| 反向控制能力 | 她掌握跨家庭的信息模式、服务动线、真实需求和信任关系；她不拥有钱，但拥有连接和判断。 |

## 4. 男性 / 资源方完整人物卡

《候鸟太太俱乐部》的资源方不是单一男主，而是一个 **男性资源结构**。

| 类型 | 设定 | 资源 | 风险 / 错位 |
|---|---|---|---|
| 付钱的先生 | 国内企业主、金融从业者、创业者、家族二代 | 服务费、房产、学校预算、家庭大方向 | 付钱但不理解海外日常；想远程控制家庭。 |
| 公司老板 | 女主所在服务公司的创始人 | 客户池、品牌、合同、提成 | 把女主的信任劳动转化成销售额。 |
| 本地资源男 | 海外地产、学校顾问、医疗陪同、生活管家类角色 | 本地通道、人情、应急资源 | 可能通过女主接近客户资源。 |
| 缺席的父亲 / 丈夫 | 不在现场但决定预算和身份路径 | 钱与名义上的家庭权力 | 让太太承担实际生活压力却没有完全决策权。 |

### 核心男性资源错位

女主每天服务的是太太和孩子，但合同、预算、付款、最终决策常常掌握在先生、公司或中介链条手里。她的权力差不是“一个男人 vs 一个女人”，而是：

> 真实需求在女性手里，正式权力在男性 / 公司 / 合同系统手里。

## 5. 女性对照组 / 女性网络

| 角色 | 功能 |
|---|---|
| 梁太太，精致陪读太太 | 住豪宅、开好车、孩子上名校，但没有家庭资产控制权。她让女主看到“体面生活的不安全”。 |
| 孟太太，强势前高管 | 表面不合群，实际最懂风险管理。她推动女主建立边界。 |
| 温太太，新移民妈妈 | 刚进入海外系统，完全依赖女主，提供读者代入视角。 |
| 许太太，老江湖 | 熟悉各类服务链条，既会帮人也会利用人，是女性网络的灰色边界提醒。 |
| 女主前辈 Ada | 曾经也被客户依赖，后来因为边界崩塌离开行业。她是女主未来的警示牌。 |

## 6. 权力差动态：初始、中段、反转

## 10. 平台传播潜力

**评分：8.5/10**

# 推荐哪个先进入 3-post MVP

## 推荐先进入 MVP：**《候鸟太太俱乐部》**

原因：

1. **平台理解成本更低。** 读者天然理解太太圈、陪读、学校、房子、身份焦虑。
2. **女性网络更强。** 它能明显纠正 Original Concept Pack v0 暴露出的“替身 / 傀儡 / proxy bias”。
3. **单元式可持续性更好。** 每个家庭都是一个 case，但女主主线可以持续推进。
4. **金丝雀机制迁移更稳。** 它保留了供养、控制、身份、物质、权力差，但不依赖单一金主关系。
5. **更适合测试 playbook 的商业可用性。** 如果这个 concept 能跑通，说明 playbook 不只能生成“女主被资源方选中”的故事，也能生成女性网络和服务业视角。

### MVP 3-post 建议

| Post | 测试目标 | 核心钩子 |
|---|---|---|
| 1 | 顾问视角能否进入高净值家庭秘密 | 入学危机 + 太太要求不要告诉先生 |
| 2 | 太太圈平台感是否成立 | 下午茶圈 + 每位太太都有两套说法 |
| 3 | 女主主线是否成立 | 公司要求变现客户隐私 + 女主第一次设边界 |

# 如果保留《她替老板养云》，应如何定位为后续实验

## 定位：AI-native Canary Mechanism Special Experiment

《她替老板养云》不建议丢弃。它的价值很高，但适合作为第二轮或平行小实验。

### 为什么保留

- 它证明 playbook 可以迁移到非传统名利场。
- 它把“金笼子”从豪宅和名表转成云资源、权限和数据，非常有原创性。
- 它贴合 AI-native product / data labor / technical authorship 这些更现代的议题。

### 为什么不建议先做主 MVP

- 技术概念门槛高。
- 情绪翻译难度高。
- 标题里的“替老板”天然有 proxy bias，需要更成熟的主动性设计。
- 如果第一轮测试失败，很难判断是 concept 不行，还是表达门槛太高。

### 后续实验方式

建议做一个非常小的专项测试：

```text
AI-native 金丝雀机制验证：3-post micro-test
```

只验证三个问题：

1. 技术权限是否能产生类似“被供养 / 被控制 / 被定价”的金丝雀感？
2. 数据署名权是否能成为情绪冲突，而不只是职场纠纷？
3. 读者是否能接受“云资源 / 权限 / 期权”作为物质细节？
